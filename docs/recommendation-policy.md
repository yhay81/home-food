# Recommendation Policy

この文書は、献立、買い物、期限、健康、食費の提案基準を定義します。実際の振る舞いは `AGENTS.md` を優先します。

## 今日の料理を提案するとき

まず `data/inventory/current.yml` を確認し、次の順で候補を作ります。

1. `urgent` または `soon` の食材を使う
2. 開封済みや残り物を使う
3. 追加購入が少ない
4. 直近の食事と栄養が偏らない
5. 調理時間がユーザーの希望に合う
6. 食費が高くなりすぎない

回答では、料理名だけでなく「使う食材」「足りないもの」「今日作る理由」を短く示します。

## 買い物を提案するとき

買い物提案は、在庫と重複しないことを最優先します。

優先度:

- `high`: 今日か明日に必要、または栄養バランス上不足している
- `medium`: 近いうちに使う予定、安ければ買ってよい
- `low`: 嗜好品、余裕があれば買う

買わなくてよいものは `data/shopping/current.yml` の `do_not_buy_now` に入れます。これは重複購入を避けるために重要です。

## 期限が近いものを答えるとき

期限が近いものを答えるときは、次を区別します。

- ラベルやユーザー発言で期限が確定しているもの
- 購入日や開封日から推定しているもの
- 画像で状態が悪そうに見えるもの
- 期限不明だが長く残っているもの

推定の場合は「推定」と明示し、使い切り方法か保存変更を提案します。

## 健康観点

健康提案は、医療判断ではなく日常の食生活の支援として行います。

見る観点:

- 野菜、きのこ、海藻、豆類の不足
- たんぱく質源の偏り
- 主食・脂質・塩分の偏り
- 発酵食品や食物繊維の不足
- 外食、惣菜、間食の増加
- 直近数日の同じメニューの連続

アレルギー、疾患、服薬、妊娠・乳幼児に関わる内容は、断定せず確認を促します。

## 食費観点

食費提案では、次を優先します。

- 期限切れ廃棄を減らす
- 同じものを重複購入しない
- 余り食材を使い切る献立にする
- 安価なたんぱく源を組み合わせる
- 冷凍・作り置きでロスを減らす
- 買う量を現実的にする

価格情報を具体的に使う場合は、最新性が変わり得るため、確認できた情報源を残します。

## 回答テンプレート

必要に応じて次の形で答えます。

```text
結論:
今日のおすすめは〇〇です。

理由:
- 〇〇が期限近めです。
- 追加購入が少なく済みます。
- 直近の食事で不足していた野菜を補えます。

買うなら:
- △△ 1袋

注意:
- □□の期限は画像から読めていないため推定です。
```
